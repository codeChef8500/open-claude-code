## 3. 系统核心设计模式总览

### 3.1 AsyncGenerator 链式传递

整个响应流使用 AsyncGenerator 链串联各层：

```
Anthropic SDK stream
    └── services/api/claude.ts   (yield StreamEvent)
          └── query.ts           (yield Message | ToolResult)
                └── QueryEngine.ts (yield SDKMessage)
                      └── REPL.tsx  (消费，渲染 UI)
```

这种设计的优势：
- **背压（Backpressure）**：消费者控制生产速度
- **零复制**：消息不需要缓冲到数组中再处理
- **可组合**：每层只关注自己的变换逻辑

### 3.2 不可变状态 + 函数式更新

```typescript
// AppState 的每次更新都产生新对象（不可变）
setAppState(prev => ({
  ...prev,                          // 浅复制
  toolPermissionContext: {
    ...prev.toolPermissionContext,  // 嵌套浅复制
    mode: 'plan',
  },
}))

// DeepImmutable<T> 类型在编译时防止意外修改
// 引用相等检测在运行时跳过无变化的更新
```

### 3.3 快速路径 + 懒加载

```typescript
// cli.tsx 中的分层快速路径
if (args[0] === '--version') return printVersion()  // 0 模块加载
if (args[0] === 'bridge') return bridgeMain()        // 按需加载
// ...
const { main } = await import('../main.js')          // 最重路径最后加载
```

### 3.5 Prompt Cache 稳定性设计

```typescript
// tools.ts 中工具列表排序的目的
return uniqBy(
  [...builtInTools].sort(byName)   // ← 排序！
    .concat(mcpTools.sort(byName)), // ← 内置在前，MCP 在后
  'name',
)
// 固定排序 + 内置工具在前 = Anthropic API 的 prompt cache 断点位置稳定
// 每减少一次 cache_creation，节省约 $0.015/1M tokens（claude-3.5-sonnet 价格）
```

### 3.6 编译时死代码消除

```typescript
// 通过 bun:bundle 的 feature() 函数实现
if (feature('BRIDGE_MODE')) {
  // 此块在外部构建中完全删除
  await import('./bridge/bridgeMain.js')
}

// 使用限制：feature() 只能用在 if/ternary 条件中，不能赋值给变量
// 因为 bun:bundle 静态分析的是语法结构，不是运行时值
```

## 4. 性能关键路径

### 4.1 启动时间优化

| 优化技术 | 节省时间 | 实现位置 |
|---------|---------|---------|
| 快速路径分发 | ~500ms | `cli.tsx` |
| MDM 并行读取 | ~200ms | `main.tsx` 副作用 |
| Keychain 预读 | ~65ms | `main.tsx` 副作用 |
| 遥测懒加载 | ~300ms | `init.ts` 动态 import |
| Early Input 缓存 | 0ms 感知 | `earlyInput.ts` |
| 预连接 Anthropic API | ~50ms TCP | `init.ts` preconnect |

**总计**：相比无优化启动可快约 1 秒以上。

### 4.2 LLM 调用延迟优化

| 优化技术 | 效果 | 实现位置 |
|---------|------|---------|
| Prompt Cache | 减少 80%+ 输入 token 计费 | 工具排序稳定性 |
| 推测执行 | 工具调用提前启动 | `SpeculationState` |
| 系统提示缓存 | 不变部分不重传 | `buildEffectiveSystemPrompt` |
| 并行记忆加载 | 系统提示与消息处理并行 | `QueryEngine.submitMessage` |

## 8. 最值得学习的设计亮点

### 亮点一：多层上下文压缩策略
从最轻量（工具结果截断）到最重量（LLM 摘要），形成渐进式压缩策略，在保留信息完整性的同时尽量延长上下文可用长度。

### 亮点二：并行预加载 + 快速路径
通过副作用并行化和快速路径分发，将冷启动时间降到最低，体现了对 CLI 工具用户体验的深度关注。

### 亮点三：类型安全的不可变状态
`DeepImmutable<AppState>` 在编译时就防止意外状态修改，而不是依赖运行时检查，零开销地提供安全保证。

### 亮点四：Tombstone 优雅降级
流式 fallback 时通过 Tombstone 消息优雅清理孤立的 thinking blocks，避免 UI 脏状态和后续 API 错误。

### 亮点五：编译时死代码消除
`feature()` + `bun:bundle` 的组合实现了真正的编译时 DCE，外部构建产物中完全不存在 ant-only 代码，既保护隐私又缩小产物大小。

## 9. 潜在改进方向（技术债务）

| 问题 | 影响 | 可能方案 |
|------|------|---------|
| `REPL.tsx` 895KB 单文件 | 维护困难，代码导航差 | 按功能区域拆分组件 |
| `query.ts` 1730 行循环 | 逻辑复杂，测试难 | 提取状态机类 |
| `ToolUseContext` 30+ 字段 | 上帝对象，耦合高 | 按关注点分组 |
| `bootstrap/state.ts` 50+ 字段 | 增长不受控 | 强制分类审批 |
| `useTypeahead.tsx` 212KB | 最大 hook，职责混杂 | 拆分为子 hooks |
| memoize 缓存的 `init()` | 测试间状态污染 | 提供测试重置机制 |
